Deterministic High-Volume MLOps Implementation Plan
Date: 2026-03-14
Scope: High-volume production MLOps while maintaining strict determinism
Determinism contract: Bit-exact within a pinned runtime (same OS, container image, hardware class)
Topology target: Single-node CPU scale-up first
Orchestration target: Celery + Redis + Postgres

====================================================================
1) Objectives
====================================================================
- Preserve strict deterministic behavior for training and inference under production load.
- Move long-running workload off synchronous API paths.
- Increase throughput via queue-backed workers and workload partitioning.
- Add production observability, retry safety, and release gates that enforce determinism.

Out of scope for this phase:
- Multi-node distributed training.
- Cross-hardware bit-exact reproducibility.

====================================================================
2) Phase Roadmap
====================================================================

Phase 1: Determinism Hardening Baseline (blocking)
- Add explicit ordering to all training-critical DB reads.
- Remove implicit iteration-order assumptions in in-memory aggregations.
- Enforce deterministic tie-breakers in ranking and run-selection paths.
- Make run persistence failure-safe with explicit state transitions.
- Add deterministic-safe artifact feature claim/order handling.
- Expand determinism tests for ordering/race/repro cases.

Phase 2: Async Orchestration Foundation
- Introduce Celery + Redis execution for training and feature extraction.
- Keep sync APIs for compatibility; add async enqueue/status endpoints.
- Add task lifecycle tracking and run-task linkage.
- Add idempotency keys for enqueue endpoints.
- Split queues by workload class (training, extraction, batch inference).

Phase 3: Dataflow and Storage Scale-up
- Convert ingestion and extraction to chunked/batched processing.
- Add deterministic checkpoint/resume behavior.
- Add DB indexes aligned to deterministic order-by paths and high-volume filters.
- Strengthen feature-cache version pinning and invalidation policy.

Phase 4: Operations and Governance
- Add queue and worker health metrics, backlog depth, latency, and saturation telemetry.
- Add structured logs with correlation IDs from request -> task -> run.
- Add CI release gate: determinism parity + integration + performance smoke.
- Add deterministic incident runbooks and replay workflow.

Phase 5: Controlled Cutover
- Canary route a subset of heavy jobs to async workers.
- Compare parity and throughput against sync baseline.
- Gradually ramp queue traffic by workload class.
- Decommission synchronous heavy-job path after sustained SLO + parity success.

====================================================================
3) Workstream Breakdown
====================================================================

A. Deterministic Data Access
- Enforce stable query ordering for entities/events/interactions/artifacts.
- Ensure deterministic ordering in latest model-run selection.
- Add stable tie-break ordering in score sorting and ranking output.

B. Deterministic Aggregation and Feature Assembly
- Replace dictionary-iteration dependent loops with sorted key traversal.
- Ensure deterministic event ordering using timestamp + stable ID tie-breakers.
- Ensure deterministic artifact vector aggregation order.

C. Persistence Correctness
- Use explicit run lifecycle states (pending, success, failed).
- Ensure DB state and filesystem artifacts cannot silently diverge.
- Define cleanup/recovery behavior for partial failures.

D. Queue-backed Execution
- Add Celery workers and Redis broker.
- Add enqueue + status APIs.
- Add task metadata, retries, and idempotency controls.
- Separate queue classes for predictable throughput and isolation.

E. Throughput and Capacity Controls
- Configure worker concurrency per queue.
- Add backpressure thresholds and queue depth alarms.
- Add bounded retries and dead-letter handling policy.

F. Deterministic Release Governance
- Gate promotions on deterministic parity suite.
- Verify model hash/metrics parity under pinned runtime.
- Block release if parity or ordering invariants fail.

5) Verification Matrix
====================================================================
Determinism parity
- Same data + same config + same pinned runtime => same run_id, model hash, metrics, and deterministic prediction subset.

Ordering invariants
- Repeated execution under load keeps stable entity/event/interaction/artifact ordering behavior.

Queue correctness
- Duplicate enqueue with same idempotency key results in one effective execution.

Failure atomicity
- Crash between artifact persistence and DB update leaves recoverable/non-ambiguous run state.

Retry determinism
- Transient retry does not alter output for identical canonical payload.

Throughput SLO
- Backlog drains within target windows while preserving determinism checks.
